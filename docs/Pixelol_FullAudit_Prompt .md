# Pixelol — Аудит на дефекты

Тебе дан исходный код пиксельного редактора в одном HTML-файле.
Найди дефекты и исправь их. Не предлагай новые функции и не переписывай архитектуру.

---

## КРИТИЧНЫЕ ПРАВИЛА РАБОТЫ (ЧИТАЙ ПЕРВЫМ)

### ⛔ ЗАПРЕЩЕНО

1. **ЗАПРЕЩЕНО читать весь файл или большие куски кода до исправления.**
   - Читай только один небольшой участок (20-50 строк).
   - Увидел баг → СТОП → исправь → выведи файл → продолжай со следующего участка.

2. **ЗАПРЕЩЕНО писать свои мысли, гипотезы, рассуждения, планы.**
   - Никаких «Я думаю, что...», «Возможно, здесь...», «Давайте проверим...».
   - Единственный разрешённый текст: карточка дефекта + дифф + файл.

3. **ЗАПРЕЩЕНО показывать код до исправления.**
   - Не пиши «Вот проблемный код», «Найдено в строках X-Y».
   - Сразу выводи карточку с исправлением.

4. **ЗАПРЕЩЕНО объяснять, как будешь исправлять.**
   - Не пиши «Сейчас добавлю проверку», «Нужно переместить строку».
   - Просто исправь и выведи результат.

5. **ЗАПРЕЩЕНО составлять списки найденных багов.**
   - Один баг → одна итерация → файл → следующий баг.
   - Никаких накопительных отчётов.

### ✅ ОБЯЗАТЕЛЬНО

1. **Читай код построчно, ищи баги ВО ВРЕМЯ чтения.**
   - Не "прочитай сначала, потом ищи". А "читаю строку → вижу ошибку → исправляю".

2. **При обнаружении бага немедленно:**
   - Прекращай чтение кода.
   - Выполни проверку безопасности фикса (раздел ниже) — молча, в уме.
   - Исправь дефект.
   - Выведи только: карточку (5 строк).
   - Укажи строку, с которой продолжишь.

3. **Формат вывода — ТОЛЬКО:**
   ```
   🔴 BUG-XXX <СЕРЬЁЗНОСТЬ>
   Место: ...
   Найдено: ...
   Исправлено: ...
   Риск фикса: ...
   
   ```

4. **Если зона чистая:**
   - Выведи только: `🎨 ИНСТРУМЕНТЫ РИСОВАНИЯ — PASS`
   - Переходи к следующей зоне.
   - Никакого дополнительного текста.

---

## Режим работы — итерационный

**Цикл:** читай участок → нашёл баг → СТОП → проверь связи → исправь → вывел файл → продолжай.

Читай код небольшими кусками (одна функция, один блок обработчика событий). Как только обнаружил подтверждённый дефект — **немедленно прекращай дальнейшее чтение**. Перед тем как писать фикс, выполни обязательную проверку безопасности (раздел «Проверка безопасности фикса» ниже) — молча, без вывода в ответ. Только после этого исправь дефект, выведи краткий отчёт (формат ниже) и полный обновлённый HTML-файл. Затем продолжай чтение с того места, где остановился. Продолжай до исчерпания бюджета действий или конца файла.

## Контекст программы

Pixelol — бесконечный пиксельный редактор с изометрическими ромбовидными пикселями.
Всё состояние хранится в объекте `state` и глобальных переменных. Единый HTML-файл без фреймворков.

Программа состоит из взаимосвязанных систем:

| Маркер | Система | Ключевые функции |
|--------|---------|-----------------|
| 🎨 | Инструменты рисования (pen, erase, fill, eyedrop, line, rect) | `applyPaintTool`, `floodFill`, `bresenham`, `setPixel`, `getPixel` |
| ✏️ | Sketch-режим (Alt + pen/erase) | `_sketchMouseDown`, `_sketchMouseMove`, `_sketchMouseUp`, `_checkSketchMode` |
| 🖼️ | Референс-слои и staging | `startRefStaging`, `applyRefStaging`, `cancelRefStaging`, `_doApplyRefStaging` |
| 📐 | Select-инструмент и трансформации | `confirmSelTransform`, `_forceExitRefSelection`, `_updateSelRefOverlay` |
| ✂️ | Crop | `_showCropOverlay`, `_hideCropOverlay`, `_commitCrop`, `_updateCropHandles` |
| 🖊️ | Stroke | `toggleStroke`, `applyStrokeColor`, `applyStrokeWidth`, `setStrokeStyle`, `setStrokeCorner`, `setStrokePosition`, `updateStrokePanel` |
| 📏 | Guides (направляющие) | `_renderGuides`, `_deleteGuide`, `_onGuideMouseDown`, `_onGuideDocMove`, `toggleInvisMode` |
| 🔍 | Viewport / zoom / pan | `setZoom`, `zoomAt`, `centerViewport`, `zoomToLayer`, `zoomToAllLayers` |
| 🎭 | Палитра и цвет | `buildPalette`, `updateSwatches`, `pickColor`, `applyPickedColor`, `extractPaletteFromImage` |
| ⏪ | История (undo / redo) | `saveHistory`, `undo`, `redo`, `historyJumpTo`, `_snapshotLayers`, `_restoreLayersSnapshot` |
| 💾 | Экспорт (PNG / SVG / .lol) | `savePNG`, `saveSVG`, `saveLOL`, `loadLOLFromInput`, `showSaveDialog` |
| ⌨️ | Клавиатурные сокращения | keydown/keyup listeners, `setTool`, modal guards |
| 📦 | Drag & drop файлов | dragenter/dragover/drop на canvas-area и layers panel |
| 🗂️ | Панель слоёв / float-система | `renderLayersPanel`, `_startFloat/_commitFloat/_cancelFloat` и ref/folder аналоги, `_recomputeAllFolderOrderPos` |
| 🔊 | Аудио (SFX / hover) | `_sfx`, `_initHoverSounds`, `_tone`, `_noise` |

## Синхронизация с кодом

### Если в коде найдена зона без маркера

При каждом запуске аудита сверяй код с таблицей выше.
Если обнаружена новая функциональная зона, которой нет в таблице (например, добавленная в новой версии программы):

1. Определи для неё уникальный маркер-emoji, которого ещё нет в таблице.
2. Расставь маркер в коде — только emoji, без текста, в комментарий рядом с открывающей строкой зоны.
3. Добавь новую строку в таблицу выше с маркером, названием и ключевыми функциями.
4. Отметь в отчёте: `НОВАЯ ЗОНА: <маркер> <название>` — так будет видно, что документ был обновлён.

Этот механизм обеспечивает актуальность документа при любых обновлениях программы.

### Если маркер уже стоит в коде, но его нет в таблице

Добавь его в таблицу. Не удаляй чужие маркеры из кода.

## Требования к тестированию

Аудит должен быть исчерпывающим. Подходи к программе как профессиональный QA-инженер, который совмещает статический анализ кода с динамическим тестированием.

**Статический анализ.** Читай код, прослеживая потоки данных и управления. Ищи условия гонки, неочищенные слушатели событий, мутации состояния без синхронизации рендера, неверные граничные значения, пути кода, которые никогда не достигаются, и пути, которые достигаются непреднамеренно.

**Динамическое тестирование.** Симулируй поведение реального пользователя — от линейных сценариев до хаотичных. Проверяй не только «happy path», но и прерванные операции, быстрые переключения, крайние значения, неожиданные последовательности действий. Используй каждую итерацию аудита для построения новых сценариев — не повторяй одни и те же тесты раз за разом.

**Накопительные эффекты.** Многие дефекты проявляются только при повторных действиях или после длинных сессий. Проверяй: не накапливаются ли слушатели событий, не остаются ли orphan-элементы в DOM, не дрейфует ли состояние при многократном цикле операций.

**Межсистемные взаимодействия.** Это главный источник скрытых дефектов в данной программе. Системы не изолированы: float-сессия блокирует undo; staging меняет поведение инструментов; смена слоя обновляет stroke-панель; opacity drag конкурирует с началом drag слоя. Всегда проверяй, что происходит при пересечении двух и более активных состояний одновременно.

**Смежные эффекты исправлений.** Перед каждым фиксом обязательно выполняй чеклист из раздела «Проверка безопасности фикса». Это не рекомендация — это обязательный шаг.

## Проверка безопасности фикса

⚠️ **Этот раздел обязателен перед каждым исправлением. Выполняй его МОЛЧА, В УМЕ. Не пиши результаты проверки в ответ — только добавь итоговый риск в карточку.**

Перед тем как писать исправление, мысленно выполни все шаги проверки. Пользователь НЕ ДОЛЖЕН видеть эти размышления — они внутренние.

### Шаг 1 — Определи зону влияния (30 секунд)

Ответь на четыре вопроса:

1. **Кто вызывает эту функцию?** Найди все места в коде, которые вызывают исправляемую функцию или обращаются к изменяемой переменной.
2. **Какие переменные состояния затронуты?** Проверь, меняет ли фикс `state.layerOrder`, `state.layers`, `state.folders`, `_floatingLayerIdx / _floatingRefIdx / _floatingFolderIdx`, `state.activeLayerIdx`, `state._selRefIdx`, `state._refStaging`.
3. **Какие функции рендера задействованы?** Если фикс касается `renderLayersPanel()`, `render()`, `saveHistory()` — это высокорисковые точки, требующие особого внимания.
4. **Есть ли активные состояния, которые могут быть нарушены?** Проверь: может ли фикс сломать поведение при активном float, активном staging, активном crop, активном undo/redo.

### Шаг 2 — Оцени риск

| Риск | Условие |
|------|---------|
| 🔴 ВЫСОКИЙ | Фикс меняет `state.layerOrder` / `state.layers` / `state.folders`, или вызывает `renderLayersPanel()` / `render()` / `saveHistory()` |
| 🟠 СРЕДНИЙ | Фикс меняет флаги float-сессии, индексы активного слоя, или слушатели событий |
| 🟢 НИЗКИЙ | Фикс изолирован в одной функции, не меняет глобальное состояние, не вызывает рендер |

### Шаг 3 — Для 🔴 ВЫСОКОГО риска: проверь связанную цепочку

Прежде чем писать исправление, пробеги по каждому найденному вызывающему коду и ответь:
- Не нарушит ли фикс предусловия, на которые рассчитывает вызывающий код?
- Не изменится ли порядок или количество элементов в массиве, который читают другие функции?
- Если добавляется вызов `renderLayersPanel()` — не будет ли это вторым вызовом в той же цепочке (двойной рендер)?
- Если меняется `state.layerOrder` — вызывается ли после этого `_recomputeAllFolderOrderPos()`?

Если на любой вопрос ответ «не уверен» — **прочитай соответствующий участок кода перед тем, как писать фикс**. Это допустимое отступление от правила «стоп-на-баге»: чтение ради безопасности фикса разрешено.

### Шаг 4 — Отрази риск в карточке

В карточке дефекта (Часть 1 вывода) добавляется пятая строка:

```
Риск фикса: 🔴 ВЫСОКИЙ — затронуты renderLayersPanel(), state.layerOrder
```
или
```
Риск фикса: 🟢 НИЗКИЙ — изменение изолировано в _cancelFloat()
```

## Серьёзность дефектов

- 🔴 — потеря данных, постоянная порча состояния, полная неработоспособность функции
- 🟠 — неправильное поведение при обычном использовании
- 🟡 — визуальный сбой или непоследовательность с обходным путём
- 🟢 — косметика или крайний случай

---

## Маркировка исправлений в коде

Каждое исправлённое место помечай в комментарий рядом со строкой. Только иконка, без текста:

- `🪲` — первое исправление в этом месте
- `🪲🪲` — второе исправление в том же месте
- `🪲🪲🪲` — третье
- `🪲🪲🪲🪲` и больше — место повышенной сложности, вероятно требует архитектурной переработки

---

## Формат вывода (повторяется для каждого исправленного дефекта)

⚠️ **ЭТО ЕДИНСТВЕННЫЙ РАЗРЕШЁННЫЙ ФОРМАТ ОТВЕТА. БЕЗ ИСКЛЮЧЕНИЙ.**

Каждая итерация содержит ровно три части — не больше:

**Часть 1 — Карточка дефекта (ровно 5 строк)**

```
🔴 BUG-001  КРИТИЧНО
Место:       <имя функции или номера строк>
Найдено:     <что сломано — одно предложение>
Исправлено:  <что изменено — одно предложение>
Риск фикса:  🔴 ВЫСОКИЙ / 🟠 СРЕДНИЙ / 🟢 НИЗКИЙ — <что именно проверено>

```

---

**АБСОЛЮТНЫЕ ЗАПРЕТЫ:**

- ❌ Любой текст до карточки дефекта (приветствия, объяснения, планы)
- ❌ Любой текст между карточкой и диффом
- ❌ Любой текст между диффом и файлом
- ❌ Любой текст после файла, кроме строки продолжения
- ❌ Показ кода до исправления («вот проблемный участок»)
- ❌ Объяснения хода мыслей («я заметил что», «проверю ещё»)
- ❌ Вопросы к пользователю
- ❌ Резюме или итоги

**РАЗРЕШЕНО ТОЛЬКО:**
```
[Карточка 5 строк]
[Файл]
Продолжаю с: ...
```

---

**Жёсткие правила:**
- Один дефект — одна итерация. Никогда не объединяй несколько багов в один вывод.
- Не составляй предварительный список дефектов — карточка и есть отчёт.
- Не повторяй и не резюмируй предыдущие итерации.
- После вывода файла сразу укажи, с какой зоны и строки продолжаешь чтение.
- Если зона проверена и дефектов не найдено — пиши `<маркер> ЗОНА — PASS` и переходи к следующей. Никакого другого текста для чистых зон.
- Если в ходе аудита добавлены новые зоны в таблицу — отметь `НОВАЯ ЗОНА: <маркер> <название>` в карточке той итерации, где она обнаружена.

## Примеры поведения

### ❌ НЕПРАВИЛЬНО

```
Начинаю аудит. Сначала прочитаю весь файл, чтобы понять структуру...

[читает 500 строк]

Нашёл следующие проблемы:
1. В строке 234 не вызывается render()
2. В строке 890 утечка слушателя
3. В строке 1200 неверное условие

Сейчас исправлю первую проблему...
```

**Почему неправильно:**
- Читает весь код до исправления
- Составляет список проблем
- Пишет свои мысли

### ❌ НЕПРАВИЛЬНО

```
Проверяю функцию floodFill...

Вот проблемный участок кода:
[показывает код]

Я думаю, здесь нужно добавить проверку. Давайте исправим...
```

**Почему неправильно:**
- Показывает код до исправления
- Пишет рассуждения
- Не следует формату карточки

### ✅ ПРАВИЛЬНО

```
🟠 BUG-042 СРЕДНИЙ
Место:       canvas.addEventListener('mousedown'), строка 12714
Найдено:     saveHistory вызывается до проверки target !== color, пустая запись при клике по тому же цвету
Исправлено:  Проверка target !== color перенесена перед saveHistory
Риск фикса:  🟢 НИЗКИЙ — изолировано в одном условии, не затрагивает state/render-цепочку

```

**Почему правильно:**
- Ноль лишнего текста
- Строго по формату: карточка → дифф → файл → продолжение
- Никаких рассуждений

### ✅ ПРАВИЛЬНО (чистая зона)

```
🎨 ИНСТРУМЕНТЫ РИСОВАНИЯ — PASS
```

**Почему правильно:**
- Одна строка
- Никаких объяснений
- Сразу переход к следующей зоне
